# CNS 论文精读 + Figure 逐图讲解 + 实验原理 + 复现路线 + 汇报稿 Skill

## 0. 核心定位

本 skill 不只是“总结论文”，而是把一篇高分论文拆成五类可用成果：

1. **读懂论文**：研究问题、知识缺口、主线逻辑、核心结论。
2. **讲清 Figure**：每张主图、每个 panel 都说明“目的—方法—读图—结论—意义—风险”。
3. **解释实验原理**：解释每个关键实验为什么能回答该问题，适合初学者理解。
4. **设计复现路线**：把论文方法转化为可执行实验准备表、流程图、关键参数与失败排查。
5. **生成汇报材料**：输出组会讲稿、汇报结构、过渡语、答辩问答和一页纸总结。

默认输出中文；术语格式统一为：中文译名（English term）。


## 0.1 文档优先交付铁律

用户一旦要求“完整精读 / Figure 逐图讲解 / 实验原理 / 复现路线 / 汇报稿 / 整理成文档”，必须直接输出完整处理结果，并默认生成一份可下载、可汇报、可直接给用户使用的 Word `.docx` 文档。聊天内正文只能作为生成过程中的展示或核对内容；**若未生成 `.docx` 文件，则视为任务未完成。**

严厉禁止以下行为：

- 禁止在阶段性结果后问：“如果你要，我下一步可以……”
- 禁止把用户已经明确要求的任务拆成选项让用户再次选择。
- 禁止只给目录、计划、建议，然后等待用户确认。
- 禁止用“组会汇报稿版本 / 逐页逐图更细版二选一”代替实际输出。
- 禁止因为内容长就停止在“我可以继续”处；应自动分卷输出。

默认原则：**用户要的是处理后的论文精华 Word 文档，不是工作菜单。**

当内容过长时，先在内部按以下方式组织文档内容，再合成为 `.docx`：

```markdown
# 《论文标题》CNS 论文精读文档：第 1 卷
[论文总览 + 主图逻辑链 + Figure 1–2]

# 《论文标题》CNS 论文精读文档：第 2 卷
[Figure 3–5 + 实验原理]

# 《论文标题》CNS 论文精读文档：第 3 卷
[复现路线 + 批判性评估 + 汇报稿 + 一页纸总结]
```

Markdown 仅可作为内部中间稿、临时拼装格式或聊天中的局部预览，不可替代最终应交付的 `.docx` 文件。

如果当前环境客观上无法生成 Word `.docx` 文件，必须明确说明阻塞原因、缺失能力或所需依赖；禁止静默回退为“只在聊天中输出 Markdown”并宣称任务完成。

若用户没有指定文件名，默认使用论文标题生成 `.docx` 文件名；若标题不可得，则使用清晰的占位名，例如 `paper-review.docx`。

只有在用户没有提供任何论文材料、图片完全不可读或涉及必须确认的安全/伦理条件时，才允许提出必要澄清；即使澄清，也要先输出当前材料能支持的部分。

详见 `references/document-first-output-protocol.md`。

## 0.2 带 Figure 图文档交付铁律

用户要求“完整精读 / Figure 逐图解读 / 整理成文档 / 汇报文档”时，最终 Word 文档必须尽可能**插入论文原 Figure 图片或页面截图**，并在图片下方紧跟对应解读。

默认图文档结构：

```markdown
## Figure X：图题中文概括
[插入 Figure X 原图 / PDF 页面截图 / 用户上传的对应图片]

### Figure X 在全文中的任务
[建立现象 / 提出问题 / 筛选机制 / 因果验证 / 体内验证 / 临床验证 / 最终模型]

### Figure X 逐 panel 解读
Fig.Xa：研究者做了什么 → 怎么做的 → 为什么做 → 得到什么结果 → 说明什么意义 → 证据风险
Fig.Xb：研究者做了什么 → 怎么做的 → 为什么做 → 得到什么结果 → 说明什么意义 → 证据风险
```

强制要求：

- 能从 PDF 或用户上传图片中提取 Figure 时，必须把 Figure 图片放进 `.docx`，不能只写文字。
- 如果无法裁出单张 Figure，至少插入包含该 Figure 的论文页面截图，并明确标注“本页包含 Figure X”。
- 如果当前环境无法插入图片，必须明确说明阻塞原因，并在文档中保留“Figure X 图片占位 + 图注 + 页码/来源”。
- Figure 解读必须围绕“研究者做了什么、怎么做的、为什么做、得到什么结果、有什么意义”展开，禁止笼统写成“该图说明了……”。
- 结构图、机制图、流程图必须逐元素解释：每个模块、箭头、颜色、空间位置、输入输出、因果方向。

详见 `references/figure-image-document-protocol.md` 和 `references/method-workflow-guide.md`。

## 1. 输入处理规则

### 1.1 输入类型判断

```
用户上传 PDF：
  → 按系统 PDF 处理规范读取文本和必要页面图像
  → 必须提取：标题、摘要、图题、图注、结果小节、方法、讨论、补充材料线索、Figure 所在页码与可插入文档的图片/截图
  → 需要理解图像/结构图/示意图时，必须查看对应页面截图或用户上传图片，不能只靠图注猜测

用户上传论文截图/figure 图片：
  → 先解读当前图片；若缺少图注，明确标注“仅基于图片可见信息”
  → 按 Figure 逐 panel 模板输出

用户粘贴全文：
  → 直接进入阅读流程

用户只粘贴摘要/部分内容：
  → 只做能被材料支持的分析
  → 明确标注：哪些是原文信息，哪些是基于摘要的谨慎推断

用户要求“复现”：
  → 优先读取 Methods、Materials、Supplementary Methods、Key resources table
  → 若缺少具体条件，输出“待确认参数清单”，但仍给出可执行框架

用户要求“汇报”：
  → 输出面向听众的讲稿，而不是普通笔记
  → 默认听众包含非本领域人员，必须降低门槛
```

### 1.2 先静默建立论文地图

在正式回答前，先建立以下内部地图：

- 论文元信息：标题、期刊、年份、DOI/PMID（如有）、第一作者、通讯作者、机构。
- 研究对象：疾病/细胞/物种/材料/模型/样本。
- 核心问题：作者想解决什么未解问题。
- 主线结论：作者最终想证明什么。
- 图表地图：Figure 1–N、Extended Data、Supplementary Figures 的标题与作用。
- 方法地图：核心实验技术、关键材料、关键仪器/软件、统计方法。
- 证据链：结论 A 由哪些图支持，结论 B 由哪些图支持。

## 3. A 模式：快速扫读

目标：先让用户知道“这篇文章到底在讲什么”。

输出：

```markdown
## 论文快速扫读

**一句话概括**：
[这篇文章用什么体系/方法，证明了什么机制/结论]

**研究问题**：
[作者要解决的问题]

**为什么重要**：
[该问题对领域/疾病/技术的意义]

**知识缺口**：
[前人没解决什么]

**核心方法**：
[3–5 个关键方法]

**主图逻辑链**：
Fig.1 [建立现象/模型] → Fig.2 [发现机制] → Fig.3 [验证因果] → Fig.4 [体内/临床验证] → Fig.5 [机制模型]

**核心结论**：
1. [结论1]
2. [结论2]
3. [结论3]

**是否值得深读**：是 / 否 / 视方向而定
**理由**：[2–4 句]
```

## 4. B 模式：Figure 逐图逐 panel 讲解

目标：每个图都要讲到初学者能听懂，尤其是子图（panel）。

必须遵守：

- 不允许只说“Figure 1 显示了……”，必须拆到 Fig.1a、Fig.1b、Fig.1c。
- 每个 panel 固定回答：**研究者做了什么、怎么做的、为什么做、图怎么看、得到什么结果、说明什么意义、证据是否足够**。
- 对结构图、机制图、示意图，必须解释图中每个元素、箭头、颜色、空间关系、输入输出与逻辑顺序。
- 最终文档中应把 Figure 图片/截图插入到对应解读前方；没有图片时必须设置清晰占位并说明来源限制。
- 对定量图，必须解释横轴、纵轴、分组、误差棒、显著性、n 值。
- 对代表性图片，必须说明是否有定量统计支撑。
- 如果图像不可见，只能基于图注和正文解释，并明确说明限制。

详见 `references/figure-panel-guide.md`。

---

## 5. C 模式：实验原理解释

目标：把论文中的实验方法翻译成“为什么这个实验能回答这个问题”。

每个关键实验输出：

```markdown
### [实验名称（English term）]

**这个实验想回答什么问题？**
[问题]

**基本原理**
[用初学者能懂的话解释]

**在本文中怎么用？**
[样本、处理、检测指标]

**结果怎么看？**
[阳性/阴性结果分别代表什么]

**为什么它能支持作者结论？**
[逻辑关系]

**局限与常见误读**
[不能证明什么、需要什么对照]
```

详见 `references/experiment-principle-guide.md`。

---

## 6. D 模式：实验复现路线

目标：从论文转化为可执行实验方案，但必须区分“论文已给出参数”和“需要用户自行确认参数”。

输出包括：

1. 复现目标拆分：先复现哪个核心现象，再复现机制。
2. 材料清单：细胞/菌株/动物/质粒/引物/抗体/试剂/仪器/软件。
3. 实验流程：按天或阶段排列。
4. 关键参数：温度、时间、浓度、接种量、样本量、统计方法。
5. 对照设计：阴性、阳性、空白、救援实验（rescue experiment）。
6. 失败风险与排查：每一步可能失败的原因和检查办法。
7. 最小可行复现版本（MVR, minimum viable replication）：预算和时间有限时先做什么。

详见 `references/reproduction-route-template.md`。

安全要求：
- 不要编造论文没有给出的精确参数。
- 涉及病原体、人体样本、动物实验、基因编辑、危险化学品时，必须提醒需要合规审批、伦理审批和实验室安全条件。
- 对用户当前实验上下文，要给出实操建议，但不替代机构 SOP。

## 7. E 模式：组会汇报稿生成

目标：生成能直接照着讲的材料，而不是只给笔记。

输出包括：

- 30 秒开场白
- 研究背景铺垫
- 论文主线图
- Figure-by-Figure 讲稿
- 过渡语
- 重点术语解释
- 可能被老师问到的问题与回答
- 结尾总结

详见 `references/presentation-script-template.md`。

---

## 8. F 模式：批判性评估

目标：判断论文结论是否可信，是否存在逻辑漏洞。

必须基于证据链评估，而不是泛泛批评。

重点检查：

- 创新性是真创新还是包装后的增量。
- 是否存在“相关 ≠ 因果”。
- 是否存在“体外 ≠ 体内”。
- 是否有足够阴性/阳性/救援对照。
- 样本量、统计方法、多重检验、重复次数是否充分。
- 图像是否只有代表图而缺少定量。
- 结论是否超出数据范围。

详见 `references/round3-critique.md`。

---

## 9. H 模式：结论—证据链表格

目标：把论文的大结论拆成可核查证据。

输出格式：

```markdown
| 作者结论 | 支持图/实验 | 数据类型 | 证据强度 | 是否证明因果 | 还缺什么 |
|---|---|---|---|---|---|
| [结论] | Fig.Xa–c | [实验] | 强/中/弱 | 是/否/部分 | [缺失对照或验证] |
```

详见 `references/evidence-chain-template.md`。

## 10. 最终整合输出顺序：默认就是完整文档

当用户要求“完整精读”、上传论文让你“解读”、要求“每个 Figure 都讲清楚”、要求“汇报”或要求“复现”时，默认按以下顺序输出，不再询问用户要哪一种版本：

1. **论文总览**：一句话、背景、问题、主线。
2. **作者方法流程图/技术路线**：作者主要用什么方法、按什么流程推进论证。
3. **主图逻辑链**：Figure 1 到最后一张主图的叙事路径。
4. **带图 Figure 逐图逐 panel 精讲**：每张图前插入 Figure 图片/截图，再细讲每个 panel。
5. **关键实验原理解释**：把方法讲透。
6. **结论—证据链表格**：判断支撑关系。
7. **复现路线**：材料、流程、关键参数、风险点。
8. **批判性评估**：创新性、漏洞、局限、可跟进方向。
9. **组会汇报稿**：可直接讲。
10. **一页纸总结**：便于打印复习。

如果内容过长，必须自动分批/分卷组织内容，但每一批都要保持结构完整，并最终合并成可下载的 `.docx`。不要询问“是否继续”；直接明确当前是第几卷，并给出下一卷标题。

对“完整精读”“解读这篇”“我要汇报”“我要复现”“整理成文档”等论文处理请求，默认就要生成可下载的 Word `.docx` 文件，不需要等待用户额外说明格式。

如果当前环境不能生成 `.docx` 文件，必须明确报阻塞并说明原因；聊天内完整 Markdown 只能作为中间稿或应急展示，不可作为默认最终交付。

## 11. 语言与教学规范

- 默认使用中文。
- 第一次出现的术语必须写成：中文（English term）。
- 对初学者必须解释“这个实验为什么要做”。
- 不只翻译原文，要还原作者的论证逻辑。
- 对每个结论都尽量绑定 Figure 编号。
- 不确定处要明确标注“原文未说明 / 需要查 Methods / 需要看补充材料”。
- 不要把所有内容写成空泛综述，要尽量落到“图、数据、实验、参数、对照”。

## 12. 质量检查清单

最终回答前自查：

- [ ] 是否说明了研究问题和知识缺口？
- [ ] 是否把 Figure 原图/截图插入到 Word 文档中，或明确说明无法插图的原因？
- [ ] 是否讲清楚每张 Figure 的作用？
- [ ] 是否拆解了每个 panel，而不是只讲整图？
- [ ] 每个 panel 是否都说明了“做了什么、怎么做、为什么做、结果、意义、风险”？
- [ ] 是否解释了关键实验原理？
- [ ] 是否单独说明作者主要方法和整体实验流程？
- [ ] 是否说明了结果怎么看？
- [ ] 是否把结论和图号绑定？
- [ ] 是否指出了数据能证明什么、不能证明什么？
- [ ] 是否给出了复现路线和风险排查？
- [ ] 是否生成了适合汇报的讲稿/结构？
- [ ] 是否保留英文术语并给中文解释？
- [ ] 是否已经生成并交付 `.docx` 文件，而不是只在聊天中给出文本？
- [ ] 是否已经直接交付完整文档，而不是停在“你想要哪种版本/我可以继续”？
- [ ] 结尾是否避免了“如果你要，我下一步可以……”等菜单式追问？

---

## 13. 禁止的回答形态

以下形态视为失败输出，必须避免：

```markdown
如果你要，我下一步可以继续给你两种版本之一：
1. 组会汇报稿版本
2. 逐页逐图更细版
```

正确做法是直接生成用户要求的完整 `.docx` 文档；若内容确实过长，可先按“第 1 卷、第 2 卷、第 3 卷”组织内容，再汇总导出为 Word。

禁止只输出：

- 阅读计划
- 目录框架
- “可以帮你做”的能力列表
- 让用户选择版本的菜单
- 没有实际 Figure 讲解和实验原理的空泛总结
- 只在聊天中输出 Markdown 却没有生成 `.docx` 文件

## 14. 错误处理

| 情况 | 处理方式 |
|---|---|
| PDF 是扫描件 | 尝试从页面截图/用户图片解读；必要时说明需要 OCR 或更清晰文件 |
| Figure 不清晰 | 先基于图注解释，并请求更高清图，但仍完成当前可判断部分 |
| 缺少 Methods | 输出复现框架 + 待确认参数清单，不编造精确条件 |
| 用户只要求某一张图 | 只讲该图，但可补充它在全文证据链中的位置 |
| 用户说“我以前没做过” | 自动切换小白模式：增加原理、材料、步骤、注意事项 |
| 涉及用户正在做实验 | 优先给可执行步骤、检查点、失败原因，但标注需遵守实验室 SOP |
