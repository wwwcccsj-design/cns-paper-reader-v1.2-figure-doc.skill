# 作者主要方法与研究流程拆解模板

## 使用场景

当用户要求“作者主要用什么方法做的 / 流程是什么 / 技术路线是什么 / 这篇文章怎么做出来的”时使用。完整精读文档也必须默认包含本模块。

## 输出目标

把论文从“结果堆叠”还原成研究者真正的工作流，让读者知道作者是如何一步一步推进问题的。

## 一、作者总体技术路线

```markdown
## 作者主要方法与研究流程

**一句话概括作者怎么做**：
作者以[研究对象]为核心，先用[方法1]发现/建立现象，再用[方法2]筛选/定位关键因素，随后用[方法3]做因果验证，最后通过[方法4]进行体内/临床/功能验证。

**总体流程**：
1. 建立问题/模型：[做了什么]
2. 发现现象：[用什么方法看到什么变化]
3. 筛选关键因素：[用什么组学/筛选/比较方法]
4. 验证机制：[敲除/过表达/抑制/救援/相互作用实验]
5. 功能验证：[细胞功能/动物模型/临床样本]
6. 形成模型：[把证据整合成机制图]
```

## 二、方法—目的—输出表

```markdown
| 方法 | 作者怎么做 | 为什么用这个方法 | 输出了什么数据 | 支持哪张 Figure/结论 | 局限 |
|---|---|---|---|---|---|
| [方法名] | [样本、处理、检测] | [回答什么问题] | [图像/表达量/功能指标/统计结果] | Fig.X | [局限] |
```

## 三、实验推进逻辑

必须按论文实际逻辑解释：

- **从现象到机制**：作者先观察到什么异常，再追问原因。
- **从相关到因果**：作者如何从表达变化/相关性，推进到敲除、过表达、救援实验。
- **从体外到体内**：作者是否用动物模型、类器官、临床样本或真实样本验证。
- **从单点证据到证据链**：每种方法补上了哪一块证据。

## 四、研究流程图文本版

```markdown
研究对象/模型
  ↓
处理或分组
  ↓
表型观察/初筛
  ↓
候选机制筛选
  ↓
关键分子验证
  ↓
因果实验：敲除/过表达/抑制/救援
  ↓
功能实验/体内实验/临床验证
  ↓
机制模型与结论
```

## 五、给初学者的解释

最后必须补一句：

> 这篇文章的方法设计本质上是在回答三个层级的问题：第一，现象是否存在；第二，是什么机制导致它；第三，改变这个机制是否会改变最终表型。

然后结合具体论文替换为实际内容。
